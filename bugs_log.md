### 1. خطأ الرفع السحابي لـ Cloudflare R2
* **الخطأ المكتشف ورسالته (Error Message):** `Access to fetch at ... has been blocked by CORS policy: Response to preflight request doesn't pass access control check: No 'Access-Control-Allow-Origin' header is present on the requested resource.`
* **الملف المتسبب وسطر الكود (Location):** [ttsService.js](file:///d:/Mypro/wasf/src/services/ttsService.js) (دالة الرفع `uploadFileToR2` السابقة).
* **الجذر التقني للمشكلة (Root Cause):** الرفع المباشر للملفات والصور من متصفح العميل (Client-side) إلى Cloudflare R2 يتعطل بسبب سياسة الـ CORS المفروضة سحابياً، ولتجاوزها بأمان كنا سنضطر لإبراز مفاتيح Cloudflare الحساسة والإدارية في المتصفح، مما يشكل خطورة أمنية بالغة.
* **طريقة الإصلاح والمعالجة (Resolution):** الاستغناء عن Cloudflare R2 في الرفع المباشر، والانتقال الكامل إلى **Supabase Storage** الموثق مسبقاً وتأسيس حاوية `wasf-assets` وتفعيل الـ RLS عليها للرفع الآمن بدون تسريب أي مفاتيح وسيطة وبدون مشاكل CORS.
* **حالة الأمان والـ RLS بعد الإصلاح (Security Status):** آمن 100%. الرفع يتم تحت مظلة مصادقة Supabase وسياستها الأمنية الصارمة.

### 9. خطأ تعليق المتصفح عند استدعاء Edge Function (JWT + CORS Preflight)
* **المشكلة:** بعد نشر Edge Function مع `verify_jwt = true`، يعلق المتصفح لدقيقتين عند الضغط على "توليد صوتي" دون أي نتيجة أو خطأ مرئي.
* **الجذر التقني:** بوابة Supabase ترفض طلب `OPTIONS` (CORS preflight) لأنه لا يحمل JWT → المتصفح يعلق في الانتظار ولا يرسل الطلب الفعلي أبداً. بالإضافة لذلك، استخدام `fetch` المباشر بدل `supabase.functions.invoke` الرسمي يعقّد معالجة الترويسات والمصادقة.
* **الحل المنفذ:**
  1. إعادة نشر Edge Function بـ `--no-verify-jwt` مع إضافة فحص أمني داخلي يتحقق من `Authorization` أو `apikey` header.
  2. استبدال `fetch` المباشر بـ `supabase.functions.invoke` الرسمي الذي يعالج CORS والمصادقة تلقائياً.
  3. إضافة سجلات تشخيصية `[TTS]` في Console لتتبع كل خطوة.
* **الدرس المستفاد:** Edge Functions المستدعاة من المتصفح يجب نشرها بـ `--no-verify-jwt` لأن بوابة JWT ترفض طلبات CORS preflight.
